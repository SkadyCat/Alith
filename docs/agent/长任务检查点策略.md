# 长任务上下文检查点策略

## 核心规则

**每完成约 20% 的上下文消耗，必须执行以下操作：**

1. 调用 `POST /open/submit` 将检查点写入 docs 服务（覆盖更新）
2. 检查点文件名格式：`任务名/checkpoint`
3. 检查点内容格式见下方模板

## 检查点模板

```markdown
# [任务名] 进度检查点
最后更新: [时间]
上下文进度: ~XX%

## 已完成
- Phase 1: [做了什么] ✅
- Phase 2: [做了什么] ✅

## 当前状态
[当前阶段、已完成内容、遇到的问题]

## 下一步计划
1. [具体步骤]
2. [具体步骤]

## 关键发现/数据
[重要文件路径、数据结构、API、已知问题]
```

## 触发时机（按上下文进度）

| 进度 | 操作 |
|------|------|
| 20% | 写检查点（Phase 1 完成总结） |
| 40% | 写检查点（Phase 2 完成总结） |
| 60% | 写检查点（Phase 3 完成总结） |
| 80% | **触发自动重置链路**（见下方） |

## 80% 自动重置流程

当上下文进度 ≥ 80% 时，立即执行：

1. **Step R-1**：读取 history/pob.md 当前内容
2. **Step R-2**：将内容压缩为极简摘要（≤ 3000 字符）
3. **Step R-3**：用压缩摘要覆盖写入 `history/应用助手4.md`
4. **Step R-4**：输出回忆确认（关键事项列表）
5. **Step R-5**：调用 `POST /agent/relaunch`，系统自动重启 Agent

## 恢复方法

新会话开始时：
1. 读取最新检查点：`GET /api/file?path=任务名/checkpoint.md`
2. 恢复上下文，继续执行下一个 Phase
